# 新聞情緒量化第一階段設計

## 目標

把目前僅靠標題正負關鍵字、輸出三種態度的新聞情緒功能，改為可解釋的輕量多因子分數。

第一階段只改善現有 Google News RSS 分析與顯示，維持 Cloud Run 1GB RAM、低冷啟動與 LINE webhook 回應速度。後續再加入每日快照、選股排序與歷史回測。

## 第一階段範圍

- Google News RSS 每則新聞保留標題、連結、來源與發布時間。
- 清理空白、來源尾碼及完全重複的標題。
- 以金融正負詞、否定詞及事件類型計算每則新聞分數。
- 依時間、來源完整度與事件重要性加權。
- 輸出 0～100 分、五級情緒、正負比例、新聞數量及可信度。
- LINE 顯示精簡摘要，Web 顯示各項拆解。

## 第一階段不做

- 不抓取 Twitter、PTT、論壇或公司公告。
- 不新增斷詞器、FinBERT、BERT、向量資料庫或背景 worker。
- 不呼叫 Gemini 判斷每則新聞。
- 不把即時情緒直接加入 LightGBM 或回填歷史資料。
- 不做模糊文字比對；第一階段只移除正規化後完全相同的標題。

## 現況問題

目前 `get_news()` 只回傳最多五則新聞的 `title` 與 `link`，發布時間與來源在解析時被丟棄。`analyze_sentiment_detail()` 則以正負詞命中次數直接加減固定分數，因此存在以下問題：

- 「不看好」可能同時命中正向詞「看好」。
- 一般評論與財報、下修、違約等重大事件權重相同。
- 剛發布與數日前的新聞權重相同。
- 多家媒體轉載同一事件時會重複計分。
- 三級標籤無法反映情緒強度與資料可信度。

## 資料流程

```text
Google News RSS
  -> 解析標題、連結、來源、發布時間
  -> 正規化與完全重複標題去除
  -> 金融詞組、否定詞、事件類型判斷
  -> 時間權重 × 來源權重 × 事件權重
  -> 加權平均與可信度
  -> LINE 精簡摘要 / Web 詳細拆解
```

RSS 欄位缺漏時保留該則新聞，但採用較低權重，不中斷整體分析。

## 單篇新聞評分

每則新聞先取得 `raw_score`，範圍限制在 `-1.0` 到 `1.0`：

1. 優先比對完整金融詞組，例如「營收創新高」、「下修財測」、「遭降評」。
2. 再計算一般正負詞命中。
3. 否定詞與後方詞組一起判斷，例如「不看好」不得計為正向。
4. 同一句同一詞組只計算一次，最後將分數限制在合法範圍。

不做中文全文斷詞。新聞標題短、金融詞組明確，先用現有字串比對即可，避免增加套件與冷啟動成本。

## 權重

### 時間權重

- 24 小時內：`1.00`
- 24～72 小時：`0.75`
- 3～7 日：`0.50`
- 超過 7 日或時間缺失：`0.25`

### 來源權重

- RSS 提供明確來源：`1.00`
- 來源缺失：`0.75`

第一階段不維護主觀媒體排名。公司公告與社群來源加入後，再分層設定權重。

### 事件權重

- 財報、營收、財測、重大政策、違約、訴訟等明確事件：`1.30`
- 一般公司或產業消息：`1.00`
- 傳聞、預測、單純看法：`0.70`

每則新聞的最終權重為三項權重相乘。

## 彙總與五級標籤

```text
weighted_score = sum(raw_score × weight) / sum(weight)
sentiment_score = 50 + 50 × weighted_score
```

最終分數限制在 0～100，初期採固定區間：

- 75～100：極度偏多
- 60～74：偏多
- 40～59：中性
- 25～39：偏空
- 0～24：極度偏空

正面比例以 `raw_score > 0.1` 計算，負面比例以 `raw_score < -0.1` 計算，其餘視為中性。固定區間只用於第一階段；累積足夠歷史資料後，再用歷史分布校準。

## 可信度

可信度與情緒方向分開，避免少量新聞造成看似精確的極端分數。

```text
可信度 = 50% 新聞數量 + 30% 新鮮度 + 20% 來源完整度
```

- 高：`>= 75`
- 中：`45～74`
- 低：`< 45`

沒有新聞時回傳 50 分、中性、可信度低，不把缺資料誤判成市場中立。

## 顯示

LINE 維持一個精簡區塊：

```text
新聞情緒　偏多 68 分
12 則｜正面 58%｜負面 17%｜可信度中
```

Web 顯示：

- 五級情緒與 0～100 分。
- 正面、負面及中性比例。
- 新聞數量與可信度。
- 每則新聞的來源、時間與判定方向。

Web 不顯示內部詞典命中細節，避免介面過度複雜。

## 錯誤處理

- RSS 逾時、XML 無效或無新聞：沿用安全中性結果，不影響股票分析。
- 單則新聞缺標題：忽略該則。
- 來源或時間缺失：降低權重，不丟棄有效標題。
- 所有外部文字在 Web 顯示前維持既有跳脫處理。
- 情緒分數只作輔助資訊，不修改五日上漲機率。

## 測試

- RSS 能解析來源與發布時間，缺漏時仍可工作。
- 完全重複新聞只計算一次。
- 「看好」為正向，「不看好」不得為正向。
- 重大事件權重大於一般評論。
- 新新聞權重大於舊新聞。
- 分數始終介於 0～100，並正確對應五級標籤。
- 沒有新聞時為 50 分、中性、可信度低。
- 新聞情緒不修改模型輸出的五日上漲機率。
- LINE 與 Web 能顯示新增欄位。

## 後續階段

### 第二階段：每日快照與選股排序

沿用既有排程與 Firestore，儲存每日個股情緒、新聞數、可信度及資料日期。取得至少 30 個交易日後加入：

- 7 日情緒移動平均。
- 情緒動能。
- 情緒波動。
- 極端情緒旗標。
- 產業預測的小權重排序因子。

### 第三階段：歷史驗證與模型特徵

資料量足夠後，依時間切分訓練與測試資料，比較情緒特徵加入前後的 Accuracy、F1、Brier Score、報酬、Sharpe Ratio 與最大回撤。只有樣本外結果穩定改善，才加入正式五日預測模型。

若規則法的錯誤率仍高，再評估排程式 Gemini 判斷或在本地執行中文金融模型；兩者都不放進 Cloud Run webhook 即時路徑。
